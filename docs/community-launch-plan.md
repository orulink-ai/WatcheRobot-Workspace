# Community Launch Plan

The official community entrance is not confirmed. This plan prepares the launch without pretending any channel already exists.

## Recommended Minimum Public Setup

| Layer | Recommendation | Reason |
| --- | --- | --- |
| Durable repo Q&A | GitHub Discussions | Closest to code, issues, releases, and long-lived answers. |
| Issue tracking | GitHub Issues | Already required for bugs, feature requests, docs, hardware, and connection issues. |
| Real-time chat | TBD: Discord / Zulip / Matrix / Feishu / other | Needs owner decision based on target audience and moderation capacity. |
| Showcase | `docs/showcase.md` plus GitHub Discussions category or issues | Keeps community projects discoverable. |
| Announcements | GitHub Releases + Discussions Announcements | Avoids scattering official updates. |

## Channel Skeleton If Chat Is Approved

Use this as a starting point for Discord/Zulip/Matrix-like communities:

| Channel | Purpose |
| --- | --- |
| `start-here` | Rules, links, hardware safety notes, and support boundaries. |
| `announcements` | Maintainer-only release and roadmap updates. |
| `general` | General community discussion. |
| `help` | Setup, build, and connection questions. |
| `ideas` | Feature ideas and creator-mode concepts. |
| `showcase` | User builds, shells, animations, motion packs, and demos. |
| `events` | Livestreams, beta tests, workshops, and release events. |

## Weekly Operating Loop

1. Triage new Issues and Discussions.
2. Label new work with `.github/labels.json`.
3. Promote 1-3 suitable tasks to `good first issue`.
4. Collect recurring support questions into docs.
5. Pick showcase submissions only after license/consent is confirmed.

## Open Decisions

- Official public community entrance.
- Moderation owner and response window.
- Whether chat history should be archived into GitHub Discussions or docs.
- Showcase submission license / consent model.
