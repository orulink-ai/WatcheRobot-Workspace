# 常见误区

## 把 meta repo 当成固件源码仓库

风险：在错误位置查找、修改或提交代码。处理方式：先确认任务属于 meta repo、ESP32 仓库还是 STM32 仓库。

## 把个人环境写进项目文件

风险：提交个人串口号、绝对路径、临时日志或缓存，影响其他人。处理方式：把个人环境作为运行时参数或本地配置，不写入共享规范。

## 双 MCU 功能只提交一侧说明

风险：reviewer 无法判断协议和行为是否配套。处理方式：ESP32 和 STM32 分别提交，并在 commit body 里说明配套关系。

## 不区分源码构建烧录和 release ZIP 烧录

风险：用错入口，导致排障方向错误。处理方式：源码流程用 `watche-s3-flash-monitor`，发布包直烧用 `watche-s3-release-binary-flash`。

## 把专项 skill 的细节复制进 onboarding

风险：SOP 膨胀且过期。处理方式：onboarding 只讲入口、判断标准和风险，专项细节交给对应 skill。

## 把一次性报错沉淀成项目规则

风险：污染 SOP。处理方式：只有高复用、项目特有、阻塞关键流程的问题才沉淀；不确定事实进入 `pending-review.md`。

## 代码修改缺少验证目标

风险：改动看似完成但行为未被证明。处理方式：开始写代码前先定义最小测试或验证目标，阶段性交付时说明验证结果或未验证原因。
