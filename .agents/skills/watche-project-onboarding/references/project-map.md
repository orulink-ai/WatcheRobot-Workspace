# 项目地图

## 仓库边界

- 当前目录是 meta repo，只管理 `.agents/`、`.codex/` 模板、workspace 脚本、文档和子仓库引用。
- App 客户端代码在 `WatcheRobot_app` 仓库中，代码修改和提交应在该仓库完成。
- 桌面客户端代码在 `WatcheRobot_client` 仓库中，代码修改和提交应在该仓库完成。
- 服务端代码在 `WatcheRobot_server` 仓库中，代码修改和提交应在该仓库完成。
- ESP32 固件代码在 `WatcheRobot_esp32` 仓库中，代码修改和提交应在该仓库完成。
- STM32 固件代码在 `WatcheRobot_stm32` 仓库中，代码修改和提交应在该仓库完成。
- 涉及多个子仓库的功能，需要分别提交，并在 commit body 中说明配套提交关系。

## 新人必须理解的心智模型

- meta repo 是工作台和知识库，不是 App、桌面端、服务端或固件源码仓库。
- `.agents/skills/` 是项目沉淀给 Codex 的自动化能力和 SOP，不是普通散文文档。
- `WatcheRobot_app`、`WatcheRobot_client`、`WatcheRobot_server`、`WatcheRobot_esp32` 和 `WatcheRobot_stm32` 是独立工程，构建、运行、测试、调试、提交边界不同。
- App、桌面端、服务端和固件联调问题通常要同时考虑 BLE/WebSocket 协议、任务状态、设备在线状态、运行时配置和硬件时序。
- 双 MCU 问题通常要同时考虑 ESP32 侧协议、STM32 侧状态机、串口/总线、硬件电源和时序。

## 提交边界

- 不要把个人端口、个人路径、临时日志、本机缓存提交到 Git。
- 修改跨 App、桌面端、服务端或固件的配套功能时，不要只提交一侧却不说明另一侧依赖。
- commit 信息使用规范、详细的中文描述。
- 阶段性交付时优先调用 `stage-delivery`，按验证、文档、提交、push、PR 的顺序收尾。

## 新人导览建议

先让新人回答三个问题：

1. 主要负责 App、桌面端、服务端、ESP32、STM32、动画资源、联调测试，还是端到端链路？
2. 当前是否有硬件台架、线缆、串口工具和可烧录设备？
3. 本次目标是只读了解，还是完成一次真实启动、测试、构建、烧录或监控？

根据回答选择后续路径，不要一次性灌输所有专项细节。
