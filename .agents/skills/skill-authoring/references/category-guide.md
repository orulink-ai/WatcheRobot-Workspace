# Skill 领域分类指南

## 分类原则

优先回答一个问题：这个 skill 最终帮助用户交付什么结果？

不要优先按语言、工具或平台分类。例如，一个使用 Python 生成固件报告的 skill，如果目标是嵌入式调试，应放入 `embedded/`，而不是软件开发方向。

## 一级领域

### `_meta`

用于管理 skill 仓库本身。

适合放入：

- skill 创建规范
- skill 审核流程
- skill 发布流程
- 仓库注册表维护

### `embedded`

用于嵌入式系统、硬件相关软件和固件开发。

适合放入：

- MCU 调试
- RTOS 任务设计
- 驱动开发
- CAN、I2C、SPI、UART 等总线诊断
- 固件构建和烧录流程

### `software-engineering`

用于通用软件开发工程能力。

适合放入：

- 代码审查
- 重构规划
- 单元测试和集成测试
- API 设计
- CI 问题定位
- 代码库迁移

### `algorithms`

用于算法设计、复杂度分析和优化问题。

适合放入：

- 数据结构选择
- 图算法
- 动态规划
- 算法题训练
- 数学优化和启发式算法

### `ai-ml`

用于人工智能和机器学习相关工作。

适合放入：

- Prompt 设计
- RAG 评估
- 模型评测
- 数据集处理
- 推理链路优化

### `devops`

用于部署、运维、发布和基础设施。

适合放入：

- Docker 构建
- Kubernetes 排障
- 发布自动化
- 监控告警
- 环境配置

### `security`

用于安全设计、审计和风险治理。

适合放入：

- 威胁建模
- 安全代码审查
- 依赖漏洞处理
- 权限设计
- 密钥和敏感信息检查

### `docs`

用于文档生产和知识表达。

适合放入：

- 技术写作
- API 文档
- 架构文档
- 变更说明
- 操作手册

## 多领域 skill 的处理

如果一个 skill 同时涉及多个领域：

1. 放入主要交付结果所在领域。
2. 在 `registry/skills.yaml` 中添加多个 tags。
3. 只有当流程明显不同，才拆成多个 skill。

示例：`embedded-security-review` 如果主要审查固件安全，应放入 `embedded/`，并添加 `security` tag。
