# 阶段性交付

## 概览

用这个 skill 把已完成的实现工作整理成可交付状态。把交付当成质量门：确认范围、按 TDD 视角验证、更新文档、创建中文详细 commit、push 交付分支，并默认打开 draft PR。

用户说“阶段性收尾”“收尾”“阶段性交付”等，且没有限制范围时，视为允许 commit、push 和创建 draft PR。用户明确说“只检查”“只准备”“不要提交”“不要 push”“不要 PR”时，只准备交付说明，不执行提交和推送。

## 交付规则

- commit body、PR 描述、交付说明、风险和下一步建议默认使用中文。
- commit 标题使用 Conventional Commit：`<type>: 中文标题`，例如 `feat: 完成登录流程阶段性交付`。
- commit type 按实际改动选择：`feat`、`fix`、`docs`、`test`、`refactor`、`chore`、`build`、`ci`、`perf`。
- 技术词自然保留英文：`TDD`、`API`、`lint`、`build`、`CI`、`README`、`typecheck`、`e2e`、`unit test`。
- 当测试失败、核心行为缺少合理覆盖、文档过期或 diff 存在无法解释的大范围改动时，不要声称交付完成。
- 默认创建 commit、push 当前交付分支并打开 draft PR。
- 除非用户明确要求，不创建 ready-for-review PR，默认是 draft。
- 工作树有无法分离的无关改动、验证失败或交付范围不清时，提交/推送前停止。
- 保护用户工作。不要回退无关改动；如果它们不属于本阶段，在交付总结中单独说明。

## 工作流

1. 确认阶段边界。
   - 从对话、issue、分支名、TODO 或最近改动中识别本阶段目标。
   - 说明已完成、刻意延期和阻塞项。

2. 检查工作树。
   - 运行 `git status --short`。
   - 用 `git diff`、`git diff --stat` 和 staged diff 审查相关改动。
   - 按功能、修复、测试、文档、重构、配置、生成资源分组。

3. 按 TDD 思路验证。
   - 新功能：检查是否有描述期望行为和关键路径的测试。
   - bug fix：检查是否有不修复会失败的回归测试，或直接覆盖修复行为。
   - 重构：检查现有测试是否保护行为；触及行为缺少覆盖时补测试。
   - 如果缺少有意义的测试且在范围内，交付前补充或更新测试。
   - 如果不能添加或运行测试，把原因写为交付风险。

4. 运行验证命令。
   - 从 `package.json`、`Makefile`、`pyproject.toml`、`Cargo.toml`、`go.mod`、`README`、CI 配置或本地惯例发现命令。
   - 先跑最窄但可靠的检查，再按影响面扩大到 test、lint、typecheck、build、e2e 或平台检查。
   - 最终交付说明和 PR 描述中记录准确命令与 pass/fail。

5. 更新项目文档。
   - 检查是否影响安装、使用、API、环境变量、部署、架构、测试、排障或用户可见行为。
   - 更新合适文档，例如 `README`、`CHANGELOG`、`docs/`、API 文档、env 示例、迁移说明或测试说明。
   - 如果无需更新文档，在交付说明里写明原因。

6. 默认创建 commit。
   - 只包含当前阶段相关文件。
   - 无关用户改动保持 unstaged，并在总结中说明。
   - 使用下方模板写详细中文 commit message。
   - 如果当前在默认分支，先创建 `codex/` 前缀的交付分支。
   - stage 选定文件并执行 `git commit`，除非用户明确要求只准备。

7. 默认 push 并打开 draft PR。
   - PR 描述用中文。
   - 包含背景、改动、TDD/验证、文档、风险和下一阶段。
   - 推送到常规 remote，并针对默认分支打开 draft PR，除非用户指定 base。
   - 优先使用可用 GitHub 工具；connector 失败时用已认证的 `gh`。
   - 用户要求只准备时，只准备 PR 标题/正文，不 push 不开 PR。

8. 输出最终交付说明。
   - 如实总结当前状态。
   - 包含改动区域、测试结果、文档更新、commit/PR 状态、风险和下一步建议。

## Commit Message 模板

```text
<type>: 中文标题

本次提交完成 <阶段/模块/问题> 的阶段性交付，重点是 <核心目标>。

主要改动：
- <改动 1>
- <改动 2>
- <改动 3>

TDD / 验证：
- <新增或更新的测试覆盖>
- `<command>` 通过
- `<command>` 通过

文档：
- <更新的文档，或说明无需更新的原因>

风险与后续：
- <风险、限制或下一阶段建议>
```

## PR 描述模板

```markdown
## 背景
<为什么需要这次阶段性交付>

## 本次改动
- <改动 1>
- <改动 2>
- <改动 3>

## TDD / 验证
- <测试覆盖说明>
- `<command>` 通过
- `<command>` 通过

## 文档更新
- <文档更新项，或说明本次不涉及文档更新>

## 风险与注意事项
- <已知风险、限制或兼容性影响>

## 下一阶段建议
- <后续建议 1>
- <后续建议 2>
```

## 最终回复格式

- 说明是否已 commit、push、打开 draft PR。
- 已提交时给出 commit 标题和 hash。
- 已打开 PR 时给出 PR 状态和 URL。
- 列出验证命令和结果。
- 列出更新的文档。
- 在说“完成”之前，先说明阻塞、测试缺口或风险。
