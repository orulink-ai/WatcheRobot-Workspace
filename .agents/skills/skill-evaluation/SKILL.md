# Skill 评估

用这个 skill 把“一个 skill 是否适合项目”评估成可执行结论，而不是泛泛讨论。

## 核心原则

- `SKILL.md` 负责判断规则：什么时候用、什么时候不用、自然语言如何映射到动作、边界、风险、需要问用户的时机和回复格式。
- `references/` 负责详细背景：评分表、长清单、项目知识、失败案例、示例输出和不常加载的领域资料。
- `scripts/` 负责确定性重复动作：解析配置、构建、烧录、清理占用、生成产物、校验文件、解析日志。
- 项目仓库脚本负责团队也要直接使用的核心能力；skill 内脚本只放 agent 专用 helper 或薄封装。
- 不要为了显得完整而创建 README、安装指南、长篇背景文档或重复项目已有文档。

## 评估流程

1. 读取目标 skill 的 `SKILL.md`，只在需要时读取它直接引用的 `references/`、`scripts/` 和 `agents/openai.yaml`。
2. 用一句话说明该 skill 当前承担的职责。
3. 判断它是否适合当前项目：
   - `保留`：职责清楚，能减少重复劳动，和项目流程匹配。
   - `修改`：方向正确，但触发条件、边界、脚本化程度或项目路径需要调整。
   - `合并`：和另一个 skill 高度重叠，继续分开会让 agent 选择困难。
   - `废弃`：当前项目没有稳定场景，或只是普通知识笔记。
4. 检查 Markdown、references、scripts 和项目脚本的分层是否合理。
5. 检查它与其它项目 skill 的边界，尤其是硬件操作、构建烧录、发布、审查和收尾类 skill。
6. 输出中文结论，并给出最小修改建议。

## Markdown 与 Script 分界

优先留在 Markdown：

- 触发场景和不触发场景。
- 用户自然语言到动作的映射。
- 风险边界、人工确认点和失败时是否问人。
- 项目术语、别名、输出格式和交付标准。
- 选择不同工具或流程的判断规则。

优先放进 script：

- 超过 3 步且经常重复的命令流程。
- 容易拼错的参数组合。
- 依赖本机路径、端口、环境变量或编码差异的执行流程。
- 需要解析输出、读写配置、校验产物或合并日志的动作。
- 失败后有固定恢复动作的流程。
- 烧录、发布、删除、迁移等出错代价高的操作。

优先放进项目仓库脚本：

- 人类开发者也会直接使用的构建、烧录、发布、导入、校验入口。
- CI、其它 agent 或团队工具需要复用的能力。
- 与项目源码版本强绑定的流程。

优先放进 skill 内脚本：

- agent 使用的辅助动作，例如清理串口占用、收集上下文、生成评估报告草稿。
- 不适合进入项目正式工具链，但能减少 agent 重复手写的薄封装。

## 需要读取的参考

当需要更细的检查项或评分口径时，读取 `references/evaluation-rubric.md`。

## 输出格式

默认用中文输出，保持可讨论、可修改：

```text
结论：保留 / 修改 / 合并 / 废弃

一句话职责：
...

适配度：
...

分层建议：
- SKILL.md：...
- references：...
- scripts：...
- 项目脚本：...

边界关系：
...

建议修改：
1. ...
2. ...
```

如果用户正在逐个讨论 skill，先给结论和理由，不要一次性大改所有 skill；用户确认后再实施修改。
