# Skill 评估评分表

需要系统评估一个 skill 时使用本评分表。评分不是为了形式化打分，而是为了暴露决策依据。

## 1. 触发清晰度

- `好`：description 明确说明什么任务、什么上下文、什么文件或用户表达会触发。
- `一般`：大方向清楚，但触发词太宽，容易和其它 skill 重叠。
- `差`：像普通介绍，无法判断什么时候应该使用。

检查问题：

- 用户一句自然语言请求能否稳定触发这个 skill？
- description 是否包含“何时使用”，而不只写“它是什么”？
- 是否误覆盖了更专门的兄弟 skill？

## 2. 项目适配度

- `好`：路径、工具、硬件、工作流和项目现状匹配。
- `一般`：方向适合，但仍有旧路径、旧命名、旧脚本或不符合当前团队习惯的假设。
- `差`：来自别的项目，当前项目没有稳定使用场景。

检查问题：

- 是否硬编码了错误 checkout、用户名、盘符或旧仓库名？
- 是否符合当前项目的真实入口，而不是复制来的抽象流程？
- 是否包含当前项目已不存在的工具、接口或产物？

## 3. Markdown 分层

- `好`：`SKILL.md` 精简，保留触发、决策、流程、边界和输出格式。
- `一般`：能用，但把较长清单或背景塞进了正文。
- `差`：正文像 README，堆了大量历史、安装、示例或普通知识。

检查问题：

- 这段内容是否每次触发都必须读？
- 详细资料是否可以拆到 `references/`？
- 有没有重复项目文档或脚本注释？

## 4. Script 化程度

- `好`：重复、确定、易错的动作已经脚本化，Markdown 只负责调用入口和决策。
- `一般`：仍有多步命令靠 agent 手拼，但风险可控。
- `差`：构建、烧录、发布、清理、解析日志等高风险流程仍只写在 Markdown。

检查问题：

- 这个流程是否超过 3 步？
- 参数组合是否容易拼错？
- 是否涉及串口、硬件、发布、删除、迁移或数据写入？
- 是否需要解析输出或验证成功标志？
- 如果同一个流程下次还要跑，是否应该沉到脚本？

## 5. 项目脚本与 Skill 脚本边界

优先项目脚本：

- 团队成员会直接运行。
- CI 或发布流程会调用。
- 与源码版本、构建系统、硬件台架强绑定。
- 需要长期维护和测试。

优先 skill 脚本：

- 只为 agent 辅助决策或收集上下文。
- 是项目脚本外的一层轻量封装。
- 不应该成为团队正式工具链的一部分。

检查问题：

- 如果没有 Codex，这个脚本是否仍然有价值？
- 如果有价值，它是否应该移动到项目仓库正式工具目录？
- skill 脚本是否只是为 agent 避免重复手写？

## 6. 边界重叠

- `好`：能清楚说明它和相邻 skill 的区别。
- `一般`：有重叠，但通过描述或输出约定可以区分。
- `差`：多个 skill 都声称处理同一任务，agent 很难选择。

检查问题：

- 单独 ESP32 烧录和双 MCU 联调是否分开？
- 源码构建烧录和 release 包烧录是否分开？
- 审查、阶段收尾和发布是否分开？
- 通用 skill 创建规范和当前项目 skill 评估是否分开？

## 7. 人机交互点

- `好`：明确什么时候先问用户，什么时候可以自动执行。
- `一般`：有些隐含交互点，但不完整。
- `差`：默认自动扫描或自动选择高风险对象，容易做错。

检查问题：

- 端口、设备、环境、目标分支、发布版本不确定时是否要问人？
- 自动检测是否可能误判？
- 是否会把“没有信息”误当成“可以猜一个”？

## 8. 输出质量

- `好`：规定了最终答复要报告什么，既有结果也有关键证据。
- `一般`：能说明成功失败，但缺少标准摘要字段。
- `差`：只让 agent 执行命令，没有说明如何向用户交代。

检查问题：

- 是否要求报告使用了哪个路径、端口、版本或命令？
- 是否要求总结验证证据？
- 失败时是否要求说明失败来源和下一步？

## 推荐结论

- `保留`：触发清晰、项目适配、边界明确，能明显减少重复劳动。
- `修改`：方向正确，但需要改路径、边界、端口策略、脚本化程度或输出格式。
- `合并`：职责和另一个 skill 高度重叠，分开会降低选择稳定性。
- `废弃`：没有当前项目使用场景，或只是普通知识，没有可复用流程。
